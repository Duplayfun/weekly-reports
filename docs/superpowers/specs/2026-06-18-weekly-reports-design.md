# Weekly Reports 项目设计文档

**日期**: 2026-06-18
**状态**: 已确认

## 一、项目概述

在本地 `weekly-reports` 项目中生成两份独立的 HTML 周报：

1. **专利周报**（`patents/`）：中国最近一周公开的、与外贸玩具或可外贸销售产品相关的专利分析
2. **Amazon DE 周报**（`amazon-de/`）：德国亚马逊户外玩具、户外游戏等品类的新品榜和热卖榜排名趋势分析

数据由 AI（Claude）采集和分析，输出为 HTML 格式 Newsletter，本地保存 + GitHub Pages 在线发布。

## 二、目标读者与语言

- **读者**：主管（高层摘要）+ 产品/选品团队（详细数据参考）
- **结构**：摘要前置，详细数据后置，一份报告满足两层需求
- **语言**：全中文（专利名称、产品名称保留原文，分析内容中文化）

## 三、生成节奏

- **频率**：固定每周六生成（若用户当天没空，可延至周日或之后手动触发）
- **覆盖范围**：当周周一至生成日的数据
- **特别生成**：用户可随时要求生成当日为止的周报（如周三临时查看）
- **触发方式**：用户在对话中手动触发（方案 A — 轻量手动模式）
- **每期新文件命名**：`{年份}-W{周数}.html`，如 `2026-W25.html`

## 五、专利周报设计

### 5.1 数据来源

- CNIPA（中国国家知识产权局）公开数据
- SooPAT / 智慧芽等商业数据库可公开部分
- 关键词方向：外贸玩具、户外游戏、可外贸销售产品
- 每期数量：30-50 条相关专利，其中精选 8-10 条做详细分析

### 5.2 内容结构（从上到下）

| 序号 | 区块 | 说明 |
|------|------|------|
| 1 | 本周概要 | 一句话结论 + 关键数字，给主管快速浏览 |
| 2 | 热门关键词 | 技术/品类趋势标签云 |
| 3 | 精选专利 Top 8-10 | 带详细摘要、外贸潜力 ★ 评分（1-5）、申请人信息 |
| 4 | 完整专利列表 | 30-50 条表格形式，供团队筛选深挖 |
| 5 | 统计卡片 | 本周相关专利总数 / 高外贸潜力数 / 关键申请人数 |

### 5.3 专利条目字段

- 专利名称（中文）
- 申请人 / 受让人
- 专利类型（发明 / 实用新型 / 外观设计）
- 公开日期
- 专利号（可点击跳转原文）
- 外贸潜力评分 ★（1-5）
- 一句话分析（AI 生成）

## 六、Amazon DE 周报设计

### 6.1 数据来源

- 德国亚马逊（amazon.de）户外玩具/户外游戏分类
- 关注榜单：New Releases（新品榜）、Best Sellers（热卖榜）
- 每期各 Top 20

### 6.2 内容结构（从上到下）

| 序号 | 区块 | 说明 |
|------|------|------|
| 1 | 本周概览 | 核心发现 + 品类热度变化趋势 |
| 2 | 新品榜 Top 20 | 表格，含排名/产品名/价格/评分 |
| 3 | 热卖榜 Top 20 | 表格，含排名/产品名/趋势箭头/价格 |
| 4 | 排名变化追踪 | 热卖榜 Top 5 四周趋势（CSS/SVG 折线图或柱状图） |
| 5 | 统计卡片 | 本周新上榜数 / 新品均价 / 热卖均价 |

### 6.3 产品条目字段

- 排名
- 产品名称（德文原文 + 中文翻译）
- 价格（€）
- 评分（★）
- 排名变化（▲▼ 箭头 + 颜色：绿色上升、红色下降、灰色持平）
- Amazon 链接

### 6.4 交叉分析

- 新品榜与热卖榜的重叠产品标记
- 连续在榜周数统计

## 八、数据采集与生成流程

### 8.1 每周执行步骤

1. **用户触发**：在对话中说"生成本周周报"
2. **并行采集数据**：
   - 专利线：搜索多个平台，筛选 30-50 条，分析精选 8-10 条
   - Amazon DE 线：获取两份榜单 Top 20，记录排名变化
3. **保存原始数据**：写入 `data/` 目录，JSON 格式
4. **生成 HTML**：数据填入模板，输出到 `reports/` 目录
5. **更新归档索引**：在 `archive/index.html` 中添加本期链接
6. **部署（可选）**：git commit + push → GitHub Actions → GitHub Pages

## 九、部署方案

### 9.1 GitHub Pages

- 用户创建 GitHub 仓库
- 配置 GitHub Actions（`.github/workflows/deploy.yml`）
- 每次 push 到 main 分支自动部署
- 固定链接格式：`https://<username>.github.io/weekly-reports/`

### 9.2 本地查看

- 直接双击 HTML 文件在浏览器打开
- 所有样式内联或使用相对路径，无需服务器

## 十、数据质量要求（⚠️ 最高优先级）

### 10.1 核心原则

**数据真实性第一。宁可数据少、宁可报告短，也不能编造。**

本条适用于所有周报类型（专利、Amazon DE 及后续新增的任何周报）。

### 10.2 专利周报数据核实

- 每条专利的**专利号**必须在 CNIPA/天眼查/PatentGuru 等数据库中查证属实
- 专利**名称和内容**必须与专利号对应，不得张冠李戴
- 从新闻报道中获取的专利信息，如报道未提供完整专利号，不得列入完整列表（可在备注中说明）
- 生成前需做两遍核实：
  1. 第一遍：每条数据在原始来源中交叉验证
  2. 第二遍：随机抽查至少 30% 的数据，确认内容一致
- 每期报告末尾标注数据来源

### 10.3 Amazon DE 周报数据核实

- 产品排名、价格、评分需来自 Amazon.de 实际页面或可信跟踪工具
- 如无法获取实时数据，需明确标注数据来源和时效性
- 产品名称链接应为可验证的 Amazon.de 搜索链接

### 10.4 质量底线

- 如果当周可核实的有效数据过少（如专利不足 3 条），应如实说明，不以编造填充
- 区分「已核实」和「待核实」两类数据

## 十一、非功能需求

- **无外部依赖**：HTML/CSS 完全自包含，不加载外部字体、图标库、JS 框架
- **离线可用**：本地 `.html` 文件可直接打开查看
- **邮件友好**：Newsletter 风格，适合直接作为邮件正文或附件转发
- **数据可追溯**：每期原始数据以 JSON 保存，支持历史对比
- **归档可查**：每份周报有独立归档索引页，列出所有历史报告
